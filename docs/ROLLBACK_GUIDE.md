# Rollback Guide

## When to rollback

Initiate rollback if ANY of the following occur during or after cutover:

- Application error rate exceeds 2x baseline for more than 5 minutes
- Health checks fail on more than 20% of target resources
- Data integrity verification fails (checksum mismatch)
- Database connectivity lost from compute tier
- Critical security finding discovered in target infrastructure
- Stakeholder or incident commander calls for rollback

## Rollback decision tree

```
Issue detected
    │
    ├─ Is traffic still on source? ──Yes──→ Abort migration, keep source
    │
    ├─ Was DNS switched? ──No──→ Destroy target resources, source unaffected
    │
    └─ DNS switched, traffic on target
        │
        ├─ Was data written to target since cutover?
        │   │
        │   ├─ No → Revert DNS, destroy target
        │   │
        │   └─ Yes → CRITICAL: sync target data back to source FIRST
        │             then revert DNS, then destroy target
        │
        └─ Is source infrastructure still operational?
            │
            ├─ Yes → Safe to revert DNS
            │
            └─ No → EMERGENCY: restore source from backup first
```

## Rollback procedure

### Step 1: Notify stakeholders

Communicate that rollback is being initiated. Include:
- Reason for rollback
- Expected duration
- Impact on users

### Step 2: Revert DNS

```bash
# If using Terraform-managed DNS:
terraform apply -target=google_dns_record_set.rollback

# If using manual DNS:
# Update A/AAAA records to point back to source IPs
# Wait for TTL expiry (check with dig/nslookup)
```

### Step 3: Verify source is healthy

```bash
# Check source infrastructure health
curl -f https://source-endpoint/health
# Verify database connectivity
# Check monitoring dashboards
```

### Step 4: Handle data written to target

If any data was written to the target after cutover:

1. Identify which databases/buckets received writes
2. Export delta from target (rows created/modified after cutover timestamp)
3. Import delta into source
4. Verify data integrity

### Step 5: Destroy target resources

Use the rollback plan's destroy order (reverse topological):

```bash
# Generated by RollbackPlanner — destroys dependents before dependencies
terraform destroy -target=google_compute_instance.app
terraform destroy -target=google_sql_database_instance.db
terraform destroy -target=google_compute_firewall.sg
terraform destroy -target=google_compute_subnetwork.subnet
terraform destroy -target=google_compute_network.vpc
```

### Step 6: Verify source state

```bash
cd source-terraform/
terraform plan  # Should show no changes (no drift)
```

### Step 7: Clean up

- Remove temporary migration resources (VPN, peering, sync agents)
- Re-import source Terraform state if it was modified
- Restore DNS TTLs to normal values
- Close the incident and document lessons learned

## Post-rollback checklist

- [ ] Source infrastructure fully operational
- [ ] All traffic confirmed flowing through source
- [ ] Monitoring/alerting active on source
- [ ] Target infrastructure fully destroyed (no orphan resources = no cost leakage)
- [ ] Data consistency verified on source
- [ ] Incident report filed with root cause and remediation plan
- [ ] Stakeholders notified of rollback completion
- [ ] Migration timeline revised based on findings

## Cost of rollback

Rollback is not free. Costs include:
- Target cloud resource charges from creation until destruction
- Data transfer (egress) charges for any data synced back
- Engineering time for the rollback procedure
- Business cost of extended downtime or degraded service

The RollbackPlanner's `estimated_duration_minutes` and the FinOps Engine's `total_migration_cost` provide pre-migration estimates of these costs.
